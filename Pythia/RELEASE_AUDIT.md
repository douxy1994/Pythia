- 2026-07-13: Hardened the previously local-only Windows x64 release workflow before its first fork run. - 2026-07-13: Verified the cross-platform automatic WebDAV schedule end to end in code. The due callback enters the real history-sync path on both platforms. Swift Core passed 17 tests, Flutter analysis reported no issues, all 72 Flutter tests passed, native Windows contracts passed, and the macOS Xcode Debug build succeeded. Live cross-device WebDAV verification still requires a reachable account and Windows x64 runtime.

Baidu uses MD5 signing, Youdao uses v3 SHA-256 signing and truncation, secrets stay in Windows Credential Manager, service order remains persisted, and portable restore retains all supported canonical provider IDs while ignoring only unknown future IDs. Provider, settings, and portable-backup tests are included in the 72-test Flutter suite; live provider verification still requires Windows x64.
- 2026-07-13: Unified macOS backup/restore with the Windows `PythiaPortableBackup` schema. Added shared Swift Codable models and strict codec validation for schema version, product identity, ISO-8601 dates, and the sensitive-fields-omitted invariant; Core now has 17 passing tests. macOS local export now writes `Pythia-backup.json` with canonical provider IDs, portable settings, and cross-platform history only. New WebDAV backup moved from `/pot` to `/Pythia/settings/portable-backup.json` and now uses temporary PUT plus MOVE with final PUT/cleanup fallback; old `/pot/pythia-config-backup.json` and `/pot/pot-config-backup.json` remain restore candidates. Windows restore retains canonical provider IDs supported on Windows, ignores only unknown future IDs, and guarantees at least one service remains selected. `PortableBackup.swift` was added to the Xcode app target; the project plist and Xcode Debug build passed. `./script/build_and_run.sh --verify` installed and launched the stable-signed `/Applications/Pythia.app`; `/config` returned `OK`, `POST /translate` returned `你好`, and `codesign` reports `Authority=Pot Local Code Signing`.
- 2026-07-13: Implemented real Windows local and WebDAV backup/restore. Added a versioned portable allowlist format and JSON Schema containing translation settings plus cross-platform history. API keys, WebDAV account data, hotkeys, launch/window state, sync status, and notifications are structurally excluded. Local export/import uses registered `file_selector_windows` native dialogs and a 64 MB limit. Restore validates fully before mutation, creates a pre-restore history backup, merges through `PythiaHistoryMerger`, preserves device settings, and marks records pending sync. Six focused tests cover omission, preservation, merge, malformed/foreign rejection, remote upload fallback, download, and missing remote data. Flutter analysis and all 68 tests passed; native dialogs and live WebDAV remain pending Windows x64 verification.
- 2026-07-13: Replaced Windows selection translation's clipboard-only behavior with a UI Automation-first path. The native runner gets the focused element, requests `UIA_TextPatternId`, reads all ranges from `IUIAutomationTextPattern::GetSelection`, trims empty selections, and returns UTF-8 without modifying the clipboard. Controls that do not expose TextPattern still use simulated `Ctrl+C`; the fallback now requires `GetClipboardSequenceNumber` to change, fixing the prior bug where an unchanged old clipboard value could be mistaken for selected text. - 2026-07-13: Closed three Windows WebDAV reliability gaps. Added a tested 10-second local-history-change scheduler used by add/favorite/delete/clear operations. Replaced immediate native tray shutdown with `app.quitRequested`/`app.quit`: Dart cancels pending debounce, waits for the current or final single-flight sync, then posts native close. - Settings now has separate OCR, TTS, and 生词本 service-instance lists; imported or installed recognize/tts/collection plugins are visible, enableable, and reorderable in their corresponding pages.
- The plugin management page now lists all plugin types instead of only translate plugins, shows the type in the popup title, and resolves the real legacy plugin directory by type.
- Saved plugin selection is restored by plugin identifier/represented object, so display names with type suffixes do not break selection restoration.
- Plugin configuration `needs` are read from the selected plugin's real type directory instead of assuming `translate`.
- Plugin configuration values from `needs` are stored locally in `plugin-configs.json`; the app no longer reads from or writes to macOS Keychain.
- Built-in API keys, proxy/WebDAV passwords, and plugin configuration values are stored locally on this machine, but config export/backup continues to omit these sensitive fields.
- Xcode Debug/Release target signing now uses the persistent `Pot Local Code Signing` certificate instead of ad-hoc signing, so macOS Accessibility/TCC grants are tied to a stable designated requirement instead of a per-build cdhash.
- Pythia network requests use an ephemeral session with persistent credential storage disabled, preventing URLSession from consulting system Keychain credentials.
- Proxy settings now affect built-in translation and WebDAV `URLSession` requests directly, including `noProxy` exclusions and optional `Proxy-Authorization`, while plugin child processes inherit `no_proxy`/`NO_PROXY` without exposing the proxy password in environment variables.
- Mixed Chinese/English input now follows the selected target language: English targets force a Chinese source, Chinese targets force an English source, and this effective pair is shared by built-in services and plugins.
- The app font, app font size, and fallback font settings now affect source/result text views. Fallback font is applied through an AppKit cascade list instead of being merely stored.
- The former transparent-window preference is now a safe Liquid Glass intensity preference and does not lower the NSWindow alpha.
- Multi-service translation now applies a per-service timeout so a hanging plugin cannot block the whole translation flow; plugin and long-form requests get longer text-aware timeouts, and the first successful service result is returned to HTTP callers.
- Plugin connection tests are type-aware: translate, recognize, TTS, and collection plugins now execute their corresponding legacy runner path instead of pretending non-translate plugins were tested by the translate path.
- `.potext` install feedback now tells the user that non-translate plugins can be managed in their corresponding settings page.
- WebDAV backup includes settings plus translation history.
- WebDAV restore writes `portable-backup.json` under `/Pythia/settings/` and retains legacy `/pot/pythia-config-backup.json` plus `/pot/pot-config-backup.json` as read-only fallbacks.
- WebDAV connectivity testing and backup upload are handled by the backup service, so the settings UI no longer owns request construction or HTTP execution details.
- Local config import and WebDAV restore share one service-layer import path that skips API keys, provider secrets, proxy/WebDAV passwords, and plugin config payloads unless an explicit sensitive-field import path is used.
- Hotkey recorder fields capture real key combinations instead of accepting arbitrary text.
- Hotkey registration now includes selection translation, input translation, screenshot translation, and screenshot OCR.
- Hotkey parsing supports letters, digits, Space, Tab, Return, Esc, arrows, and F1-F12.
- Collection plugins now have real user-facing entry points in the main translator window footer, the macOS View menu, and the status-item menu. Legacy collection plugins receive the current source text plus successful translation result payload through `POT_TARGET`.
- History window actions now validate selection, confirm destructive operations, show success/failure feedback, and export as `pythia-history.json`.
- Settings history export uses the same `pythia-history.json` naming and reports the exported record count.
- The "关闭历史记录" setting now has a real storage-layer effect: when enabled, `HistoryStore.add` returns without mutating `history.json`, so main-window, HTTP, hotkey, clipboard, and selection translation flows all share the same history-off behavior.
- HTTP `/translate` now reuses the main translation flow and returns the first successful service result without issuing a duplicate service request or writing duplicate history.
- Pure model/error definitions now live in `Models/PythiaModels.swift`, the app entrypoint/application delegate and menu construction live under `App/`, persistent stores live under `Stores/`, reusable AppKit controls and `HistoryWindowController` live under `Views/`, and plugin management, legacy Pot configuration migration, backup/config import, built-in translation/network dispatch, HTTP routing, OCR, selected-text reading, clipboard monitoring, plus hotkey monitoring now live under `Services/`, reducing the original monolithic app file.

## Known Gaps To Continue Auditing

- The app is still partly implemented in oversized AppKit controller files; model definitions, `PythiaAppDelegate`/`@main`, native menu construction, `HistoryStore`, `Preferences`, `PluginManager`, `MigrationService`, backup/config import, built-in `TranslationService`, `SelectionReader`, `OCRService`, `PythiaHTTPServer`, `ClipboardMonitor`, `HotKeyMonitor`, reusable controls, and `HistoryWindowController` have been split out, but `TranslatorWindowController.swift`, `SettingsWindowController.swift`, and remaining UI-specific builders should continue moving into smaller dedicated files before release.
- Liquid Glass usage should be reviewed visually on the target macOS version; current implementation relies mostly on AppKit visual effect views and custom drawing.
- Original Pot plugins that call Tauri `run_binary` are reported as unsupported.
- OCR, TTS, and collection plugin instances are now presented as manageable services and have type-aware synthetic runtime tests from the plugin settings page. OCR plugin execution still needs deeper validation against real third-party recognize plugins before it can fully replace the built-in OCR path.
- End-to-end manual UI testing is still needed for custom hotkey recording, screen capture permissions, Accessibility selection reading, and interactive screenshot OCR region selection.
